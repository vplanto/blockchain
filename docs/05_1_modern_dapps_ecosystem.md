## 1. Сучасні стандарти розробки (Modern Development Standards)

З розвитком екосистеми Ethereum підходи до розробки значно змінилися. В сучасному production-середовищі варто враховувати такі аспекти:

### 1. Layer 2 (L2) рішення та масштабованість
Оскільки комісії (газ) у головній мережі Ethereum (L1) стали занадто дорогими для звичайних користувачів, більшість сучасної розробки та деплою DApps відбувається на мережах другого рівня (таких як **Arbitrum**, **Optimism**, **zkSync**, **Base**). Важливо розуміти:
- Код на Solidity компілюється та працює на L2 абсолютно так само, як і на L1 (завдяки EVM-сумісності).
- Комісії за транзакції там коштують долі центів, тоді як на L1 можуть сягати десятків доларів.

### 2. Професійні фреймворки (Foundry та Hardhat)
**Remix IDE**, який ми використовували раніше, є ідеальним інструментом для навчання або швидкого прототипування. Однак у реальних проектах архітектура розгортається локально за допомогою спеціалізованих фреймворків:
- **Foundry** — сучасний галузевий стандарт (написаний на Rust). Дозволяє писати складні тести безпосередньо на Solidity, має неймовірну швидкість компіляції та виконання тестів.
- **Hardhat** — популярний JavaScript/TypeScript фреймворк, який дозволяє розгортати локальні форки (копії) реальної мережі Ethereum для тестування контрактів зі збереженням стейту.

### 3. Оновлювані смарт-контракти (Proxy Patterns)
За замовчуванням деплой контракту незмінний — після відправлення у блокчейн його логіку не можна переписати. Щоб обійти це обмеження і дозволити виправлення багів (оновлення коду), сучасний стандарт вимагає використання **Proxy-патернів** (наприклад, Transparent Proxy або UUPS). 
- Створюються два контракти: `Proxy` (який зберігає стан/баланси та адресу користувача) та `Implementation` (який містить логіку).
- Коли потрібне оновлення, власник просто направляє вказівник `Proxy` на нову адресу `Implementation`.

### 4. Стандарти токенів
Сучасний курс по Web3 неможливий без базового розуміння стандартів токенів (протоколів ERC), що лежать в основі DeFi та NFT:
- **ERC-20** — стандарт взаємозамінних (fungible) токенів (наприклад, USDT, стейблкойни). Всі токени однакові.
- **ERC-721** — стандарт невзаємозамінних токенів (NFT). Кожен токен є унікальним (картини, цифрова нерухомість).
- **ERC-1155** — мультитокенний стандарт (часто використовується в іграх для одночасного зберігання як унікальних, так і звичайних предметів у межах одного смарт-контракту).

## 2. Побудова dApps: Клієнтська частина та інтеграція

Смарт-контракт — це лише бекенд. Для користувача "додатком" є веб-сайт або мобільна аплікація, яка взаємодіє з блокчейном. Ця взаємодія відбувається через спеціальні бібліотеки та гаманці.

### Архітектурний патерн (Клієнтські абстракції)
Більшість сучасних dApps використовують абстракції для читання та запису даних. Існує два основних галузевих стандарти:

**1. Класичний ООП-підхід (на прикладі `ethers.js`):**
- **Provider (Провайдер):** Це "вікно" у блокчейн. Він дозволяє тільки читати дані (наприклад, баланс, стан контракту, номер блоку). Підключається до ноди через Infura/Alchemy.
- **Signer (Підписант):** Це абстракція гаманця (Account) з доступом до приватних ключів. Потрібен для будь-якої дії, що змінює стан мережі (запис).
- **Contract:** Об'єкт, що поєднує адресу контракту, його ABI та Провайдера/Підписанта.

**2. Сучасна функціональна парадигма (на прикладі `viem`):**
- **PublicClient:** Відповідає за публічні (читальні) RPC-запити до блокчейну (аналог Provider).
- **WalletClient:** Відповідає за дії, що потребують підпису транзакцій через гаманець користувача (аналог Signer).
- **Actions:** Замість створення великого об'єкта `Contract`, використовуються чисті функції (наприклад, `readContract` або `writeContract`).

### Інтеграція з гаманцем (EIP-1193)
Браузерні гаманці (MetaMask, Rabby) ін'єктують об'єкт `window.ethereum` у веб-сторінку.
- **Підключення:** `window.ethereum.request({ method: 'eth_requestAccounts' })`.
- **Зміна мережі:** Гаманець повідомляє додаток про зміну мережі або акаунту через події (events). Розробник повинен обробляти це, щоб уникнути помилок (наприклад, відправка токенів не в ту мережу).

### ABI (Application Binary Interface)
ABI — це JSON-файл, який генерується компілятором Solidity. Він описує всі функції контракту, їхні аргументи та типи повернення. Без ABI фронтенд не знатиме, як правильно закодувати виклик функції для EVM.

## Definition of Done (Критерії засвоєння матеріалу)

Лекцію вважаємо успішно засвоєною, якщо студент здатен:

- [ ] **Розрізняти стандарти токенів:** Пояснити різницю між ERC-20 та ERC-721.
- [ ] **Розуміти Proxy-паттерн:** Пояснити, як саме оновлюється логіка смарт-контракту, якщо його код незмінний.
- [ ] **Розуміти архітектуру dApps:** Розрізняти ролі `Provider` (читання/підключення) та `Signer` (офлайн підпис транзакцій/запис).
- [ ] **Знати призначення фреймворків:** Пояснити, чому для production-розробки використовується Foundry/Hardhat, а не Remix IDE.
