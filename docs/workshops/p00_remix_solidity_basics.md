# Практикум p00. Remix IDE та основи Solidity: читання коду

**Мета:** Отримати практичний досвід: відкрити приклади контрактів у Remix, скомпілювати, задеплоїти та перевірити поведінку. Уміння читати Solidity і бачити основні концепти в коді.

**Код прикладів:** папка [solidity](https://github.com/vplanto/blockchain/tree/main/solidity) у репозиторії курсу.  
**Середовище:** [Remix IDE](https://remix.ethereum.org/) (онлайн, без встановлення).

## 1. Підготовка

1. Відкрийте [Remix IDE](https://remix.ethereum.org/).
2. Створіть файл у папці `contracts` (наприклад, `HelloWorld.sol`)
3. Вставте наступний базовий код смарт-контракту в редактор:
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract HelloWorld {
    uint private value;
    
    function setValue(uint newValue) public { 
        value = newValue; 
    }
    
    function getValue() public view returns (uint) { 
        return value; 
    }
}
```

## 2. Процес Компіляції та Деплою (Deploy)

**Компіляція (Compile):**
Процес перекладу коду з високорівневої мови Solidity у низькорівневий байт-код (машинний код), який розуміє Ethereum Virtual Machine (EVM). У Remix це робиться у вкладці *Solidity Compiler*.

**Деплой (Deploy / Розгортання):**
Це процес відправки вашого скомпільованого байт-коду в блокчейн-мережу. Технічно це транзакція без отримувача (`to: null`), де в поле даних (`data`) записується код контракту. Після успішного деплою код назавжди закріплюється за новою унікальною адресою (Contract Address), і до нього можна звертатися.

**Ключові параметри в Remix:**
- **Environment (Remix VM - напр. Cancun / Osaka):** Це віртуальна, імітована блокчейн-мережа, що працює виключно у вашому браузері. Вона миттєво обробляє транзакції і має набір тестових акаунтів зі 100 "фейковими" ETH. Версії на кшталт "Osaka", "Cancun" або "Shanghai" — це кодові назви хардфорків (великих оновлень) мережі Ethereum, які емулює ця віртуальна машина.
- **Environment (Injected Provider - MetaMask):** Спосіб для розгортання контракту у справжню мережу (тестнет або мейннет). Ваш браузер з плагіном MetaMask не є повноцінною блокчейн-нодою. Замість цього MetaMask інкапсулює ваш приватний ключ і формує транзакцію, але відправляє її через Ethereum API до інфраструктурного провайдера (зазвичай це Infura або Alchemy), які вже доставляють її до мережі.
- **Value:** Сума коштів (у wei / gwei / ether), яку ви прикріплюєте до транзакції деплою або до виклику конкретної функції. Передається з вашого акаунту на баланс контракту. Важливо: якщо ви передасте `Value` > 0 у контракт під час деплою, але його конструктор не помічений як `payable`, транзакція буде відхилена.
- **Gas Limit:** Максимальна кількість газу, яку ви дозволяєте витратити на виконання цієї транзакції. Відіграє роль запобіжника. Якщо обчислення виявляться складнішими, ніж очікувалося (або станеться нескінченний цикл), після вичерпання ліміту виконання примусово зупиниться, і всі зміни відкотяться назад (Out of Gas).

### Розбір типового фейлу деплою

Припустимо, ми спробували зібрати та задеплоїти наш контракт `HelloWorld`, і в полі **Value** випадково вписали `15 wei`. У консолі ми отримали б такий лог:

```text
status	0 Transaction mined but execution failed
transaction hash	0x4e9a579f74daf3be7ffa6d36dd4e121cc063ed97694556a23c78405a3f1bae9d
contract address	0xd9145CCE52D386f254917e481eB44e9943F39138
from	0x5B38Da6a701c568545dCfcB03FcB875f56beddC4
to	HelloWorld.(constructor)
transaction cost	58217 gas 
value	15 wei

creation of HelloWorld errored: Error occurred: revert.

revert
	The transaction has been reverted to the initial state.
Note: The called function should be payable if you send value and the value you send should be less than your current balance.
```

**Як правильно читати цей лог:**
1. **`status: 0 ... execution failed`** — мережа прийняла транзакцію до блоку (`mined`), але під час виконання інструкцій сталася помилка.
2. **`to: HelloWorld.(constructor)` та `value: 15 wei`** — ми намагалися створити екземпляр контракту (через конструктор) і одночасно надіслали туди 15 wei.
3. **`creation of HelloWorld errored: Error occurred: revert.`** — EVM примусово відкотила (`revert`) всю транзакцію. Код контракту **не був** збережений.
4. **`Note: The called function should be payable if you send value...`** — Remix підказує нам точну причину. Відсутній явно заданий конструктор з модифікатором `payable`. Без цього слова EVM автоматично відхиляє будь-які перекази ефіру до контракту заради безпеки.
5. **Наслідки:** Всі витрати газу за цю невдалу спробу (`transaction cost 58217 gas`) були списані з вашого акаунта і назавжди згоріли (бо мережа виконала роботу по валідації). Самі "15 wei" залишились на балансі, оскільки транзакцію було відкочено.

### Інтерфейс взаємодії після деплою (Deployed Contracts)

Після успішного Деплою ваш контракт з'явиться внизу панелі у блоці **Deployed Contracts**. Розгорнувши його, ви побачите не лише створені вами функції, але й спеціальний системний розділ унизу картки:

- **Low level interaction (Низькорівнева взаємодія):** Спосіб звернутися до контракту напряму, оминаючи звичайні кнопки функцій. Ви можете безпосередньо відправляти `calldata` (сирі байти). Найчастіше використовується для тестування того, як контракт реагує на неочікувані виклики або звичайні перекази (тестування функцій `fallback` та `receive`).
- **Value:** Поле для вказання суми коштів (wei/ether), яку ви хочете надіслати разом з цим низькорівневим викликом прямо в контракт.
- **Gas limit:** Максимальний ліміт газу, виділений саме на цю транзакцію взаємодії. Працює як запобіжник контролю витрат.
- **auto:** Режим автоматичного розрахунку газу. Коли увімкнено, Remix перед реальною відправкою локально симулює транзакцію (робить Estimate) і "автоматично" підставляє найбільш оптимальний ліміт, що рятує від помилки `Out of Gas`.

## 3. Завдання: HelloWorld

**Код для аналізу:**
```solidity
contract HelloWorld {
    uint private value;
    function setValue(uint newValue) public { value = newValue; }
    function getValue() public view returns (uint) { return value; }
}
```

1. Вкладка **Compile**: виберіть компілятор (наприклад, 0.8.20), натисніть Compile.
2. Вкладка **Deploy & run transactions**: Environment — "Remix VM (Cancun)". Впевніться, що поле **Value** пусте (`0`). Натисніть **Deploy**.
3. У панелі викликів викликайте `setValue` з аргументом, наприклад 42. Функція змінює стан контракту, тому списує газ (це **транзакція**). Перевірте як зменшився баланс вашого аккаунта на соті долі відсотка.
4. Потім викликайте `getValue` і переконайтеся, що повертається 42. Функція лише читає стан (`view`), тому вона є **безкоштовною** для зовнішнього світу (це **виклик / call**).

**Смисловий підсумок:** Контракт не "крутиться" у фоні наче програма на ПК. Він є структурним об'єктом в пам'яті блокчейну, який "активується" лише коли до нього ініціюють транзакції.

## 4. Завдання: Voter (v1 і v2)

1. Створіть новий файл `Voter_v1.sol` та розгляньте його конструктор:
   ```solidity
   string[] public options;
   uint[] public votes;

   constructor(string[] memory _options) {
       options = _options;
       // Ініціалізація динамічного масиву лічильників тією ж довжиною
       votes = new uint[](_options.length); 
   }
   ```
   В Remix при деплої можна передати масив рядків, наприклад `["A", "B", "C"]` (синтаксис залежить від інтерфейсу).
2. Після деплою викликайте `vote(0)`, `vote(1)`, `vote(0)` кілька разів. Перевірте `getVotes()`: чи збільшуються лічильники?
3. Створіть новий файл `Voter_v2_OneVoteOnly.sol`. Задеплойте з тими самими опціями. Спробуйте двічі викликати `vote(0)` з одного й того самого акаунта. Друга спроба має завершитися помилкою (revert). Чому?
4. У коді `v2` знайдіть `mapping` та приватну функцію `recordVote`:
   ```solidity
   mapping (address => bool) hasVoted;

   function recordVote(uint option) private {
       hasVoted[msg.sender] = true;         // Запам'ятовуємо, що акаунт вже голосував
       votes[option] = votes[option] + 1;
   }
   ```
   Поясніть одним реченням, як за допомогою змінної `msg.sender` реалізовано ліміт "один голос на адресу".

**Що перевірити в коді:** `require` для валідації опції та для заборони повторного голосування; приватна функція `recordVote` і де оновлюється стан.

## 5. Завдання: Voter v3 і v4 (рядки та газ)

1. У файлі `Voter_v3_StringOptions.sol` є функція `vote(string memory option)`. Як вона знаходить індекс опції?
   ```solidity
   for (uint i = 0; i < options.length; i++) {
       string memory currOption = options[i];
       // Порівняння рядків через криптографічні хеші:
       if (keccak256(bytes(option)) == keccak256(bytes(currOption))) {
           recordVote(i);
           return;
       }
   }
   revert(); // Якщо код дійшов сюди, значить опції не знайдено
   ```
   Подивіться на цей цикл і пряме порівняння кожного елементу масиву. Це алгоритм з часовою складністю $O(N)$.
2. У покращеній версії `Voter_v4_VoteByString.sol` той самий виклик реалізований без циклу:
   ```solidity
   struct OptionPos {
       uint pos;
       bool exists;
   }
   // Словник, який зберігає зв'язок (Назва опції => Її позиція), створений у конструкторі:
   mapping (string => OptionPos) posOfOption; 

   function vote(string memory option) public {
       require(!hasVoted[msg.sender], "Already voted");
       
       OptionPos memory pos = posOfOption[option]; // Швидке отримання об'єкту
       require(pos.exists, "Invalid option");

       recordVote(pos.pos);
   }
   ```
   Чому такий підхід (зчитати зі словника за $O(1)$) набагато економніший за газ при великій кількості опцій, навіть якщо створення такого словника в конструкторі коштує дорожче?
3. У Remix після виклику функції перегляньте витрату газу (Gas used). Порівняйте (на рівні ідеї) виклик vote по індексу в v2 і vote по рядку в v3 при, наприклад, 10 опціях.

**Що перевірити в коді:** структура `OptionPos`, ініціалізація `posOfOption` у конструкторі; чому пошук за рядком у v4 — O(1).

## Питання та відповіді (екзаменаційний блок)

**П1.** Як у Remix вибрати середовище виконання і що таке "Remix VM"?

**В1.** У вкладці "Deploy & run transactions" поле "Environment" дозволяє вибрати, де виконувати транзакції. "Remix VM" — це емуляція Ethereum у браузері без підключення до реальної мережі. Корисно для навчання: безкоштовно, миттєво, стан зберігається в сесії.

---

**П2.** У контракті Voter v2 яка роль `msg.sender` і mapping `hasVoted`?

**В2.** `msg.sender` — адреса того, хто безпосередньо викликав функцію (у Remix — обраний акаунт). `hasVoted[msg.sender]` зберігає, чи вже ця адреса голосувала. Перед зарахуванням голосу перевіряється `require(!hasVoted[msg.sender])`; після голосування встановлюється `hasVoted[msg.sender] = true`. Так реалізується обмеження "один голос на адресу".

---

**П3.** Чому в Voter v4 голосування за рядком дешевше за газ, ніж у v3?

**В3.** У v3 пошук опції за рядком робиться циклом по всьому масиву `options` і порівнянням хешів — це O(n) операцій і багато читання з пам’яті. У v4 у конструкторі заповнюється mapping "рядок → позиція"; при голосуванні за рядком достатньо одного звернення до mapping — O(1). Менше операцій — менше газу.

## 🎓 Інженерний міні-кейс (на 1 бал)

Ви створили смарт-контракт голосування і задеплоїли його в мережі Ethereum. Раптом виявилось, що ви забули додати функцію `getWinningOption()`, щоб побачити, хто переміг. Однак, всі функції голосування `vote(uint index)` працюють коректно, і люди вже віддали тисячі голосів. 
**Питання:** Чи зможете ви (або будь-хто інший) дізнатися результати голосування, якщо в контракті немає функції для читання цих результатів, але стан блокчейну відкритий? Обґрунтуйте відповідь.
