# Практикум 03: Самостійні завдання на розробку смарт-контрактів

Ось 15 сучасних, нетривіальних завдань для студентів, розроблених на основі концепцій блокчейну (EVM, смарт-контракти, DeFi, DAO, оракули), адаптованих під реалії України та технічні можливості Remix IDE.

> [!IMPORTANT]
> **Золотий стандарт індустрії: OpenZeppelin**
> У жодній продуктовій блокчейн-компанії Senior інженери не пишуть стандарти токенів (ERC-20, ERC-721), логіку безпечної математики чи алгоритми контролю доступу (`Ownable`) з нуля. Смарт-контракти не прощають помилок, тому **інженерія в Web3 — це про перевикористання безпечного, перевіреного аудиторами коду**.
> 
> Для вирішення завдань (особливо тих, що стосуються токенів та управління доступом) вам *категорично рекомендується* успадковувати ваші контракти від бібліотеки **OpenZeppelin**. 
> У Remix це робиться одним рядком: `import "@openzeppelin/contracts/token/ERC20/ERC20.sol";` (Remix самостійно завантажить актуальний код із GitHub під час компіляції).

## 📋 15 Сучасних завдань для смарт-контрактів

| № | Назва завдання / Тематика | Опис бізнес-логіки (Що повинен реалізувати студент) | Ключові концепти для перевірки |
| :--- | :--- | :--- | :--- |
| **1** | **Система штрафних балів ПДР** | Створити реєстр водіїв. Поліція (адмін) може нараховувати бали за порушення. Якщо сума балів досягає 15, статус водія автоматично змінюється на "Права анульовано". Водій може перевірити свій статус (view-функція). | Модифікатори доступу (`onlyAdmin`), структури даних (`mapping`), читання стану. |
| **2** | **Квитки на Укрзалізницю проти перекупників** | Реалізувати систему продажу квитків у вигляді Soulbound NFT (токен, який неможливо передати на іншу адресу після мінту). Користувач купує квиток, але функція `transfer` повинна бути заблокована (revert). | ERC-721, перевизначення методів стандарту, `require`, `revert`. |
| **3** | **Прозорий збір на ЗСУ (Multisig фонд)** | Контракт для збору донатів (ETH). Вивести кошти можна лише тоді, коли мінімум 2 з 3 визначених волонтерів (адмінів) підпишуть (викликчуть функцію `approveWithdrawal`) запит на зняття певної суми. | Масиви/мапінги для зберігання підписів, мультипідпис (Multi-sig), безпека виведення коштів. |
| **4** | **Бронювання працівників (Timelocked Status)** | Підприємство реєструє працівника. Статус "заброньований" видається рівно на 6 місяців (моделюється через `block.timestamp`). Після завершення терміну контракт автоматично вважає статус недійсним без додаткових транзакцій. | Робота з часом у Solidity (`block.timestamp`), оптимізація газу (відсутність постійних перерахунків). |
| **5** | **Агро-страхування від посухи (Оракул)** | Фермер вносить страховий платіж. Якщо в регіоні посуха, фермер отримує виплату. Оскільки EVM ізольована від інтернету, створити функцію `updateWeather`, яку може викликати лише спеціальна адреса (Оракул). *Примітка: реалізуйте патерн Mock Oracle (наприклад, модифікатор `onlyOracle`), щоб ви могли вручну симулювати передачу зовнішніх даних через Remix.* | Ізоляція мережі (The Isolation Wall), архітектура Оракулів (Oracles), умовні виплати. |
| **6** | **Програма лояльності Сільпо (ERC-20 + Burn)** | Випуск власних токенів `Власного Рахунку`. Нарахування токенів за "покупки". Студент має реалізувати логіку: покупець може спалити (Burn) 100 токенів, щоб отримати знижку (зміна внутрішнього статусу клієнта). | Стандарт ERC-20, спалювання токенів (burn mechanics). |
| **7** | **Децентралізовані петиції (Дія.Петиції)** | Створення петиції з таймером. Кожен користувач може проголосувати лише 1 раз. Якщо за Х блоків набирається 25,000 голосів — петиція отримує статус "На розгляді". | Захист від повторного голосування (`mapping(address => bool)`), лічильники, оптимізація. |
| **8** | **DAO для ОСББ (Керування будинком)** | Мешканці будинку мають токени "голосу". Створюється пропозиція: "Ремонт даху на 5 ETH". Щоб гроші переказалися підряднику, пропозиція має набрати кворум (більше 50% токенів мешканців). | On-chain Governance, кворум, делегування голосів. |
| **9** | **Спадщина (Dead man's switch)** | Контракт-сейф. Власник зберігає там гроші і має функцію `ping()`. Якщо власник не викликав `ping()` більше ніж 1 рік, вказаний спадкоємець отримує право вивести всі кошти. | Управління станом (State Machine), контроль доступу (`msg.sender`). |
| **10** | **Спрощений AMM-обмінник (Uniswap V2)** | Створити смарт-контракт, який імітує пул ліквідності двох токенів. Розрахунок ціни обміну має відбуватися за формулою Constant Product Formula: `X * Y = K` (де X та Y — резерви токенів, а K — константа пулу). Реалізувати функцію swap із врахуванням прослизання. Підказка: `amountOut = (reserveOut * amountIn) / (reserveIn + amountIn)`. | Математика DeFi, Constant Product Formula, уникнення помилок округлення. |
| **11** | **Аукціон конфіскату (Pull over Push)** | Учасники роблять ставки в ETH. Переможець забирає лот. **Важливо:** контракт не повинен сам відправляти ETH назад тим, хто програв (щоб уникнути Reentrancy). Студент має реалізувати функцію `withdraw`, де переможені самі забирають свої ставки. | Патерн Checks-Effects-Interactions (CEI), захист від Reentrancy/DoS. |
| **12** | **Flash Loan симуляція** | Контракт видає будь-яку суму в борг без застави, за умови, що в кінці цієї ж функції борг + 1% комісії буде повернуто. Студент має написати контракт-видавач і контракт-приймач, щоб показати атомарність. *(Спирається на лекцію про DeFi: Flash loans виконуються і повертаються в межах однієї транзакції)*. | Миттєві кредити (Flash Loans), атомарність EVM (revert всієї транзакції при фейлі). |
| **13** | **Зарплатний стрімінг** | Роботодавець депонує 1000 USDT на місяць. Працівник може викликати функцію `withdraw` щодня і забирати лише ту частину грошей, яка пропорційна часу, що минув з початку місяця. | Математика з плаваючою крапкою в Solidity (через множення/ділення великих чисел), змінні стану. |
| **14** | **P2P Енергоринок (Сонячні панелі)** | Користувачі продають надлишки енергії. Контракт є ескроу-сервісом: покупець резервує ETH, оракул (лічильник) підтверджує передачу кВт, після чого ETH відправляється продавцю. | Escrow-логіка, оновлення балансів, блокування коштів. |
| **15** | **Чорні списки (OFAC / Sanctions)** | Створити токен, де спеціальна адреса (Адміністратор) може вносити інші адреси в `blackList`. Будь-які перекази з або на чорно-спискові адреси мають блокуватися (revert). | Compliance у блокчейні, `require`, робота з мапінгами. |

## 🛠 Як здавати виконані завдання

Оскільки Remix IDE працює локально в браузері, ви не можете просто надіслати текстовий "файл" на перевірку. Використовуйте один із двох підходів для здачі завдань:

### Варіант 1: Швидкий (через Remix Gist / Share URL)
1. **Здача:** Ви пишете код у Remix і натискаєте іконку створення GitHub Gist (або Share URL) прямо в IDE. Відправляєте згенероване посилання викладачу.
2. **Перевірка:** Викладач перейде за посиланням, код автоматично відкриється. Після компіляції викладач протестує логіку `require` та всі методи контракту у середовищі `Remix VM`.

### Варіант 2: Професійний (через Sepolia Testnet та Etherscan)
1. **Здача:** Ви деплоїте свій контракт у тестову мережу (наприклад, Sepolia) через Injected Provider (MetaMask). Після цього вам необхідно верифікувати код контракту на Etherscan і надіслати викладачу лише **адресу контракту**.
2. **Перевірка:** Викладач відкриває адресу на `sepolia.etherscan.io`. Переходить у вкладку `Contract -> Read/Write Contract` і бачить весь початковий код. Прямо через веб-інтерфейс Etherscan він підключає свій MetaMask та тестує роботу контракту. Цей варіант підтверджує ваше вміння працювати з реальними Web3 мережами та RPC.

## 💡 Важливі рекомендації перед стартом

> [!IMPORTANT]
> **5 правил розробника смарт-контрактів:**

1. **Немає інтернету всередині EVM:** Ви не можете написати `fetch("api.police.gov.ua")`. Блокчейн — це ізольований бункер. Всі зовнішні дані (погода, рішення суду, курс долара) мають заноситись у контракт через транзакції від довірених адрес (Оракулів).
2. **Газ коштує реальних грошей:** Уникайте циклів `for` по величезним масивам даних. Якщо вам треба знайти водія за ID, використовуйте `mapping(uint => Driver)`, який працює за $O(1)$, а не перебирайте масив, бо транзакція впаде з помилкою `Out of Gas`.
3. **Безпека понад усе (Правило CEI):** Якщо ваше завдання передбачає переказ ETH (наприклад, Аукціон або Спадщина), **завжди** спочатку оновлюйте внутрішні баланси контракту, і лише останнім рядком робіть `transfer()`. Інакше ваш контракт зламають через атаку Reentrancy (повторний вхід).
4. **Математика без дробових чисел:** Solidity не підтримує числа типу `1.5`. Для розрахунку відсотків (наприклад, комісія чи стрімінг зарплати) потрібно спочатку множити на великі числа, а потім ділити (наприклад, `(amount * 1) / 100` для 1%).
5. **Тестуйте у Remix VM:** Перед тим як деплоїти у Sepolia і чекати підтвердження блоків, відлагоджуйте логіку у середовищі `Remix VM`. Воно миттєве, безкоштовне і дозволяє легко перемикатися між 10 тестовими акаунтами, щоб імітувати взаємодію різних користувачів.

## ⭐ Бонусні вимоги (для підвищеної оцінки)

Виконання базового завдання у Remix дає стандартну оцінку. Для отримання вищого балу реалізуйте одну або обидві з наступних вимог:

### Бонус A: Покриття тестами у Foundry

Перенесіть ваш контракт з Remix у [Foundry-проєкт](./p04_foundry_local_dev.md) та напишіть до нього автоматичні тести. Мінімальний набір:

- **Unit-тест:** Перевірити, що основна бізнес-логіка працює коректно (наприклад, для завдання №1 — що бали нараховуються і статус змінюється при досягненні 15).
- **Тест контролю доступу:** Перевірити через `vm.prank`, що функції з обмеженим доступом (`onlyAdmin`, `onlyOracle`) відхиляють виклики від сторонніх акаунтів.
- **Fuzz-тест (опціонально):** Додати хоча б один `testFuzz_` для перевірки контракту на випадкових вхідних даних.

Приклад структури тесту для завдання №1 (Штрафні бали ПДР):

```solidity
// test/TrafficPenalty.t.sol
import {Test} from "forge-std/Test.sol";
import {TrafficPenalty} from "../src/TrafficPenalty.sol";

contract TrafficPenaltyTest is Test {
    TrafficPenalty public penalty;
    address police = makeAddr("police");
    address driver = makeAddr("driver");

    function setUp() public {
        vm.prank(police);
        penalty = new TrafficPenalty();
    }

    function test_AddPenaltyPoints() public {
        vm.prank(police);
        penalty.addPoints(driver, 5);
        assertEq(penalty.getPoints(driver), 5);
    }

    function test_RevertIfNotPolice() public {
        vm.prank(driver);
        vm.expectRevert(); // Очікуємо revert
        penalty.addPoints(driver, 5);
    }

    function test_LicenseRevokedAt15() public {
        vm.startPrank(police);
        penalty.addPoints(driver, 10);
        penalty.addPoints(driver, 5);
        vm.stopPrank();
        assertTrue(penalty.isRevoked(driver));
    }
}
```

### Бонус B: Інтеграція з фронтендом (читання стану контракту)

Після деплою вашого контракту у тестнет Sepolia, напишіть мінімальний скрипт на JavaScript, який читає дані з контракту через публічну RPC-ноду. Це демонструє розуміння того, як dApps взаємодіють з блокчейном.

**Приклад скрипту (ethers.js v6):**

```javascript
// read_contract.mjs
// Запуск: node read_contract.mjs
import { ethers } from "ethers";

// 1. Підключення до публічної ноди Sepolia
const provider = new ethers.JsonRpcProvider(
  "https://eth-sepolia.g.alchemy.com/v2/YOUR_KEY"
);

// 2. ABI — інтерфейс контракту (масив функцій).
//    Можна скопіювати з Remix після компіляції (вкладка ABI)
const abi = [
  "function getPoints(address) view returns (uint256)",
  "function isRevoked(address) view returns (bool)"
];

// 3. Адреса вашого задеплоєного контракту у Sepolia
const contractAddress = "0xYOUR_CONTRACT_ADDRESS";

// 4. Створення об'єкту контракту
const contract = new ethers.Contract(contractAddress, abi, provider);

// 5. Виклик view-функції (безкоштовний, не потребує підпису)
const points = await contract.getPoints("0xSOME_DRIVER_ADDRESS");
console.log("Штрафні бали:", points.toString());

const revoked = await contract.isRevoked("0xSOME_DRIVER_ADDRESS");
console.log("Права анульовано:", revoked);
```

**Альтернатива (viem):**

```javascript
// read_contract_viem.mjs
import { createPublicClient, http } from "viem";
import { sepolia } from "viem/chains";

const client = createPublicClient({
  chain: sepolia,
  transport: http("https://eth-sepolia.g.alchemy.com/v2/YOUR_KEY"),
});

const points = await client.readContract({
  address: "0xYOUR_CONTRACT_ADDRESS",
  abi: [{ name: "getPoints", type: "function",
           inputs: [{ type: "address" }],
           outputs: [{ type: "uint256" }],
           stateMutability: "view" }],
  functionName: "getPoints",
  args: ["0xSOME_DRIVER_ADDRESS"],
});

console.log("Штрафні бали:", points);
```

> [!TIP]
> Для запуску скриптів вам знадобиться: `npm init -y && npm install ethers` (або `npm install viem`). Безкоштовний RPC-ключ можна отримати на [alchemy.com](https://www.alchemy.com/) або [infura.io](https://www.infura.io/).
