## 1. Що таке смарт-контракт та що змінюється при переказі

Смарт-контракт — це програма, що зберігається в блокчейні і виконується автоматично за заздалегідь визначеними правилами (кодом).

**Звичайний переказ (Externally owned -> Externally owned):**
Змінюються лише баланси.
- Відправник (Address: 0x123): мав 20 ETH -> стає менше (наприклад, 15 ETH).
- Одержувач (Address: 0x321): мав 20 ETH -> стає більше (наприклад, 25 ETH).

**Переказ на контракт (Externally owned -> Contract):**
Змінюються баланси, а також може змінитися **стан** контракту і можуть бути викликані його **методи**.
- Відправник (Address: 0x123): `Balance: 20 ETH` -> `send money` -> стає менше.
- Одержувач (Contract Address: 0x456): `Balance: 5 ETH` -> стає більше.
- Крім балансу, змінюється стан контракту. Наприклад: `State: giveawaysLeft=0` (оновилися внутрішні змінні).
- Викликаються методи: наприклад, метод `SendMoney` (або `receive` / `fallback`).

## 3. Життєвий цикл смарт-контракту (Smart Contracts Lifecycle)

- **Написання коду**: Ви визначаєте методи (methods) та поля стану (state fields) у мові Solidity (як класи ООП).
- **Деплой (розгортання)**: З одного коду (шаблону) можна задеплоїти один або декілька екземплярів (instances) контракту в мережу.
- **Незмінність**: Після розгортання методи та поля (структура) стають статичними (non-dynamic) — додати нові методи чи змінити сутність полів неможливо.

## 4. Структура смарт-контракту (Solidity)

Ось базовий приклад структури смарт-контракту на Solidity з розбором його складових:

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// contract - ключове слово для оголошення смарт-контракту (аналог class у мовах ООП)
contract Greeter {
    
    // Стан (state variables) - зберігається безпосередньо в блокчейні
    uint private val;

    // Конструктор - викликається лише один раз під час деплою (розгортання)
    constructor(uint initialValue) {
        val = initialValue;
    }

    // Функція читання (view) - не змінює стан, при зовнішньому виклику безкоштовна (не витрачає газ)
    function getValue() public view returns (uint) {
        return val;
    }

    // Функція запису - змінює стан (val), вимагає транзакції та оплати комісії (газ)
    function setValue(uint newValue) public {
        val = newValue;
    }
}
```

- **`// SPDX-License-Identifier:`**: Вказує тип ліцензії коду (наприклад, MIT, UNLICENSED). Допомагає уникнути юридичних попереджень компілятора.
- **`pragma solidity ^0.8.0;`**: Вказує версію компілятора (у цьому випадку будь-яка версія від 0.8.0, але менша за 0.9.0).
- **`contract Greeter`**: Оголошення самого контракту, подібне до оголошення класу.
- **Змінні стану (`uint private val`)**: Змінні, значення яких перманентно зберігаються у сховищі (storage) контракту в блокчейні.
- **`constructor`**: Спеціальна функція, яка виконується рівно один раз при створенні (деплої) контракту в мережу.
- **Модифікатори видимості та поведінки (`public view`)**: Визначають, хто може викликати функцію і чи може вона змінювати стан мережі. `view` гарантує, що функція лише читає дані.

## 5. Основи синтаксису Solidity

Solidity за своїм синтаксисом схожа на C++, JavaScript та Java. Ось базові концепції мови.

### Коментарі (Comments)
- `// Однорядковий коментар`
- `/* Багаторядковий коментар */`

### Основні типи даних (Types)

```solidity
bool flag;                   // Логічний тип (true / false)
int signedNum;               // Знакове 256-бітне ціле число
uint unsignedNum;            // Беззнакове 256-бітне ціле число (рекомендується для балансів)
int8, int16, ... int256;     // Знакові цілі від 8 до 256 біт
uint8, uint16, ... uint256;  // Беззнакові цілі від 8 до 256 біт
string message;              // Рядок символів (динамічний)
address userAddress;         // 20-байтна Ethereum адреса (формат: 0x123...)
```

### Змінні (Variables) та Стан

Змінні стану (state variables) назавжди зберігаються у блокчейні. Локальні змінні існують лише під час виконання функції у пам'яті.

```solidity
contract Variables {
    uint public value;       // Змінна стану (в блокчейні)
    bool public flag;        // Змінна стану

    function setVals(uint newVal, bool newFlag) public {
        uint updatedVal = newVal + 1; // Локальна змінна (в пам'яті EVM)
        
        // Оновлення стану контракту
        value = newVal;
        flag = newFlag;
    }
}
```

### Константи (Constants)

Змінні, значення яких ніколи не змінюється і задається під час компіляції. Вони не займають місце у сховищі (storage), що заощаджує газ.

```solidity
contract Greeter {
    uint constant MS_IN_SEC = 1000;

    function toSeconds(uint timeInMs) public pure returns(uint) {
        return timeInMs / MS_IN_SEC; // Використання константи
    }
}
```

### Оператори (Operators)

**Математичні:**
```solidity
uint newVal = (a - b + c) * 2 / 4;
uint remainder = a % b;      // Остача від ділення
uint power = a ** 3;         // Піднесення до ступеня (a^3)
uint left = a << 2;          // Побітовий зсув вліво
uint right = b >> 4;         // Побітовий зсув вправо
num += 2;                    // Аналог num = num + 2;
num++;                       // Інкремент
```

**Порівняння та логіка:**
```solidity
bool larger = a > b;
bool smaller = a < b;
bool exp = (a >= b) && (a <= c);  // Логічне "І" (AND)
bool exp2 = (a == b) || (a <= c); // Логічне "АБО" (OR)
```

### Функції (Functions)

Методи контракту, що виконують логіку, змінюють стан або повертають значення.

```solidity
contract MyContract {
    uint val;
    
    // Функція, що повертає значення
    function getValue() public view returns(uint) {
        return val;
    }
    
    // Функція, що змінює стан
    function setValue(uint _val) public {
        val = _val;
    }
}
```

**Повернення декількох значень (Return Values):**
```solidity
// Декілька значень (кортеж)
function getValues(uint a) public pure returns(uint, uint) {
    return (a, a + 1);
}

// Повернення іменованих змінних (значення присвоюються автоматично)
function getValuesNamed(uint a) public pure returns(uint b, uint c) {
    b = a;
    c = a + 1;
    // return писати не обов'язково
}
```

**Виклик інших функцій (Calling Other Functions):**
```solidity
contract MyContract {
    function calc() public pure {
        sum(1, 2);              // Позиційна передача параметрів
        sum({a: 1, b: 2});      // Передача за іменами параметрів
    }
    
    function sum(uint a, uint b) public pure returns(uint) {
        return a + b;
    }
}
```

**Перевантаження функцій (Functions Overloading):**
Можна створювати декілька функцій з однаковим ім'ям, але різними типами параметрів.
```solidity
function validate(string memory v) public pure returns(bool) { /* ... */ }
function validate(uint v) public pure returns(bool) { /* ... */ }

// Виклики: validate("str"); або validate(12);
```

### Модифікатори доступу (Access Modifiers)

Оскільки контракти розгортаються у публічній мережі (будь-хто може викликати публічні методи і змінити стан), критично важливо визначати права доступу. Це аналог інкапсуляції в ООП мовах (Java, C#).

**Для функцій:**
```solidity
function accessibleFromAnywhere() public { /* ... */ }
// function accessibleFromAnywhereDefault() { ... } // У старих версіях - public, зараз вимагає явного вказання
function onlyFromOutside() external { /* ... */ }
function accessibleInternally() internal { /* ... */ }
function accessibleOnlyHere() private { /* ... */ }
```

**Для змінних стану (Field Modifiers):**
```solidity
uint public value1;   // Можна читати ззовні (автоматично створюється getter функція)
uint internal value2; // Доступно лише зсередини цього контракту та його нащадків
uint private value3;  // Доступно лише зсередини цього контракту
uint value4;          // За замовчуванням змінні стану є internal
```

### Змінні `msg` (Global Messages)
Глобальна змінна `msg` містить інформацію про поточний виклик/транзакцію:
```solidity
function method() public payable {
    msg.value;   // Кількість відправлених Wei (якщо є payable)
    msg.sender;  // Адреса того, хто безпосередньо викликав функцію
    msg.data;    // Повний payload транзакції (байт-код виклику та його аргументи)
    msg.sig;     // Ідентифікатор функції (перші 4 байти msg.data)
    // msg.gas;  // Значення залишкового газу (застарілий атрибут, зараз використовується gasleft())
}
```

### Масиви (Arrays)
У Solidity всі елементи масиву повинні бути одного типу. Елементи отримуються чи задаються за їх позицією (індексом).
- **Динамічні масиви (Dynamic Arrays):** розмір може змінюватися.
  ```solidity
  uint[] dynamic1;                     // Порожній динамічний масив у storage
  uint[] memory dynamic2 = new uint[](2); // Динамічний масив у пам'яті (розмір задається при створенні)
  
  uint i = dynamic1[5];     // Читання елемента
  dynamic1[5] = 10;         // Редагування
  uint size = dynamic1.length; // Розмір
  dynamic1.push(42);        // Додавання нового елемента в кінець (лише для storage масивів)
  ```
- **Статичні масиви (Statically-sized Arrays):** розмір фіксується при компіляції.
  ```solidity
  uint[5] arr5;             // Масив жорсткої довжини 5
  // arr5.push(42);         // ПОМИЛКА: Неможливо додавати елементи
  arr5[0] = 10;             // Дозволено
  uint size = arr5.length;  // Завжди 5
  ```
- **Інші види масивів:**
  ```solidity
  uint[][] nested;          // Багатовимірні (вкладені) масиви
  bytes1 oneByte;           // Фіксований масив байтів (від bytes1 до bytes32)
  bytes dynamicBytes;       // Динамічний масив байтів (аналог byte[])
  ```

### Типи за посиланням проти типів за значенням
- **Типи за значенням (Value types):** (базові типи `uint`, `bool` тощо) створюють незалежну копію при присвоєнні `b = a`.
- **Типи за посиланням (Reference types):** (масиви, мапінги, структури) при присвоєнні в межах одного роду пам'яті будуть посилатися на той самий об'єкт. Зміна `b[0] = 1` змінить і `a[0]`.

### Storage, Memory та Calldata
Сховища даних контракту радикально відрізняються ціною і призначенням:
- **Storage:** Перманентний стан контракту (зберігається вічно). Зміна storage-змінних дуже дорога (витрачає багато газу, до 20к `SSTORE`).
- **Memory:** Тимчасова пам'ять під час виконання функції. Після завершення виклику — очищується. Коштує дешево (від 3 газу).
- **Calldata:** Тимчасова, *незмінна (immutable)* область для аргументів зовнішніх викликів. Найдешевша пам'ять.

```solidity
contract Foo {
    uint[] arr; // Зберігається у Storage

    function method(uint[] calldata extData) public {
        uint[] memory mArr = arr;    // Копія зі сховища в тимчасову пам'ять
        mArr[0] = 0;                 // Змінює *лише копію*
        
        uint[] storage sArr = arr;   // Посилання (вказівник) на Storage
        sArr[0] = 0;                 // Змінює *оригінал* у блокчейні!
    }
}
```

### Конструктори (Constructors) та Ініціалізація
Для ініціалізації полів контракту використовується конструктор:
- Аналог ООП-мов, але смарт-контракт може мати лише **один** конструктор (відсутнє перевантаження).
- Виконується тільки один раз при розгортанні.

```solidity
contract Foo {
    string public value;
    string constant MAX_VAL = "10";   // Завжди незмінна, компілюється в байткод, не займає місця
    uint immutable maxAge;          // Модифікатор стану `immutable`: змінну можна задати лише під час розгортання в конструкторі. Потім вона "вшивається" в байткод контракту і не може бути змінена.

    constructor(string memory _value, uint _maxAge) {
        value = _value;
        maxAge = _maxAge; // Immutable-змінні встановлюються тут
    }
}
```

### Дефолтні значення (Default Values)
У Solidity не існує поняття `null` чи `undefined`. Змінні завжди мають початкове нульове значення безпечно:
- `bool flag;` ➔ `false`
- `uint num;` ➔ `0`
- `uint[] arr;` ➔ `[]` (порожній масив)
- `string str;` ➔ `""` (порожній рядок)

### Умовні оператори (If-Else) та Порівняння
Повністю аналогічно C++ / JavaScript, але умова всередині `if` має бути **строго** типу `bool`. Не можна писати `if (1)` або `if ("")`, як у JS.
**Порівняння рядків:** пряме порівняння `if (str1 == str2)` у Solidity не підтримується. Замість цього використовуйте криптографічні хеші:
```solidity
if (keccak256(bytes(str1)) == keccak256(bytes(str2))) { ... }
```

### Цикли (Loops)
Підтримуються `for`, `while`, `do..while`, а також команди `break` та `continue`:
```solidity
for (uint i = 0; i < arr.length; i++) {
    if (i == 3) continue; // Пропустити ітерацію
    if (i == 5) break;    // Вийти з циклу
    sum += arr[i];
}
```

### Структури даних (Structs)
Структури — це колекції полів (нагадують класи, але без методів і наслідування). Їх можна вільно вкладати одна в одну.
```solidity
struct User {
    string name;
    uint8 age;
}

// Ініціалізація або позиційно, або за назвою полів:
User memory user1 = User("John", 35);
User memory user2 = User({ name: "John", age: 35 });
user1.age = 36; // Зміна поля (через крапку)
```

### Мапінги, або Словники (Mappings)
Відомі також як Hash Maps, Dicts. Структура виду `mapping(KeyType => ValueType)`.
Основні переваги: неймовірна швидкість отримання елемента `O(1)` і економія газу, оскільки не потрібні дорогі `for` цикли для пошуку.
```solidity
mapping(address => bool) public hasVoted; // Хто вже голосував
mapping(string => uint) public scoreFor;  // Зберігання балів

scoreFor["Adam"] = 1;         // Запис
uint s = scoreFor["Adam"];    // Читання (1)
uint x = scoreFor["Unknown"]; // Читання неіснуючого ключа повертає дефолт (0)
```

### Валідація і Помилки (Rejecting Invalid Transactions)
Якщо користувачі намагаються викликати транзакцію з неправильними аргументами або нелігальним станом (наприклад, голосують вдруге), транзакцію потрібно негайно перервати:
1. `require(умова, "Повідомлення");` — перевіряє вхідні аргументи та логіку стану. При фейлі відкочує всі зміни стану (revert) і повертає залишок газу.
2. `revert("Повідомлення");` — безумовний відкат (наприклад, всередині блоку `if`). Найсучасніший підхід (зменшує комисії за газ) використання **Custom Errors**:
   ```solidity
   error DuplicateValue(uint pos, uint value); // Визначення нагорі
   ...
   if (invalid) revert DuplicateValue({pos: 0, value: val});
   ```
3. `assert(умова);` — сувора перевірка неймовірних станів (інваріантів) системи. Спалює весь доступний газ при помилці.
*(Примітка: старий оператор `throw;` вважається застарілим і забороненим).*

### Захист від переповнення (Integer Overflow Protection)
Комп'ютерна математика має ліміти (наприклад, змінна `uint8` тримає числа тільки від 0 до 255).
- До версії компілятора `0.8.0`: вираз `uint8(255) + 1` призводив до сайлент-"переповнення", і змінна тихо ставала рівною `0`. Тому раніше повсюдно використовували бібліотеку `SafeMath`.
- Починаючи з версії `0.8.0`: EVM автоматично викидає помилку і відкочує транзакцію при переповненні. Якщо ви хочете навмисне дозволити класичне переповнення для швидкості, використовується обгортка `unchecked { res = a + b; }`.

## 5.1. Компіляція, Байт-код та Ethereum Virtual Machine (EVM)

Вузли (ноди) Ethereum не виконують код на Solidity чи Vyper напряму. Вони вимагають і розуміють лише низькорівневий машинний код — **Ethereum Bytecode**. Це дозволяє розробляти смарт-контракти на різних мовах високого рівня, які потім компілюються в єдиний стандартний формат.

`Solidity / Vyper code` ➔ `Solidity / Vyper compiler` ➔ `Ethereum Bytecode` ➔ `Виконання у EVM`

### Команди EVM (Opcodes)
Байт-код складається з простих базових інструкцій (Opcodes).
| Команда | Байт | Опис (Description) |
| :--- | :--- | :--- |
| `ADD` | `0x01` | Додавання двох чисел |
| `MUL` | `0x02` | Множення двох чисел |
| `LT` | `0x10` | Порівняння "менше ніж" (Less-than) |
| `EQ` | `0x14` | Порівняння рівності (Equality) |
| `SHA3` | `0x20` | Обчислення хешу SHA3 (Keccak256) |
| `CREATE`| `0xf0` | Створення нового контракту |
| `SSTORE`| `0x55` | Запис слова (word) у постійне сховище (storage) |

### Стек-машина (Stack-based VM)
EVM є віртуальною машиною на основі стеку (Stack-based VM). Всі операції виконуються над верхніми елементами стеку.
- `PUSH` — додає елемент на вершину стека.
- `POP` — видаляє елемент з вершини стека.
- Команди (наприклад, `ADD` або `MUL`) автоматично беруть потрібну кількість аргументів з вершини стека, виконують дію і кладуть результат назад на вершину.

**Приклад роботи стеку для виразу: `2 * 3 + 1 = ?`**
```text
Байт-код команди:
PUSH 2  // Стек: [2]
PUSH 3  // Стек: [3, 2]
MUL     // Бере 3 і 2, множить, кладе 6. Стек: [6]
PUSH 1  // Стек: [1, 6]
ADD     // Бере 1 і 6, додає, кладе 7. Стек: [7]
```

## 5.2. Транзакції, Стан та Атомарність виконання

Блокчейн Ethereum можна уявити як машину станів (State Machine). Будь-яка зміна цього стану ініціюється виключно транзакціями.

### Як змінюється стан блокчейну (Changing Ethereum State)
Змінити стан мережі можна такими діями:
1. Задеплоїти (розгорнути) новий контракт.
2. Відправити ефір (Ether) між акаунтами.
3. Викликати метод контракту, який змінює його внутрішній стан (state).

**Процес переходу:**
`Поточний стан Ethereum` ➔ `Виконання транзакції` ➔ `Новий стан Ethereum`
Оскільки блоки складаються з транзакцій, виконання кожної з них (в рамках блоку) послідовно переводить мережу в новий консенсусний стан.

### Атомарність виконання (Atomic Execution)
Виконання транзакцій є абсолютно **атомарним**:
- Транзакція або виконується успішно цілком (всі зміни зберігаються).
- Або відкочується повністю (revert), не залишаючи жодних проміжних змін у стані смарт-контракту (хоча комісія за газ списується).

**Послідовність:**
Транзакції виконуються строго одна за одною (послідовно). Вони не працюють паралельно і не можуть непередбачувано втручатися (interfere) одна в одну під час виконання однієї інструкції.

### Структура Транзакції (Transaction Structure)
Жодна дія в мережі не відбувається без транзакції. Її складові:
- `nonce` — порядковий номер транзакції (кількість транзакцій, відправлених з цієї адреси).
- `gas limit` — максимальна кількість газу, яку ви виділяєте на виконання.
- `destination` — адреса отримувача (гаманець користувача або смарт-контракт).
- `amount` (`value`) — сума Ether, яка переказується.
- `v, r, s` — криптографічний підпис (доводить, що транзакцію підписав власник акаунта).
- `init` або `data` (payload) — скомпільований байт-код (при деплої нового контракту) або закодований виклик методу контракту з аргументами.

***Розгортання контракту (Contracts Deployment):*** Коли створюється новий контракт, транзакція відправляється без адреси `destination` (вона порожня), а поле `data` містить `init` код контракту. Мережа EVM виконує цей код і створює нову адресу смарт-контракту зі стартовим станом.

## 5.3. Механіка виконання (Methods Execution, Блоки, Обмеження)

### Поля Блоку (Block Fields)
Після того, як транзакція відправлена, вона потрапляє в мемпул, а звідти пакується валідаторами (validators) у блок. Кожен блок містить:
- **Block number** — порядковий номер блоку.
- **Parent's block hash** — криптографічний хеш попереднього блоку (що і створює "ланцюг").
- **New state of the network** — криптографічний доказ (Patricia tree root) поточного стану всіх акаунтів (баланси, стейти контрактів) після виконання всіх транзакцій цього блоку.
- **Timestamp** — час прийняття блоку.
- **Executed transactions** — повний список виконаних транзакцій.

### Обмеження смарт-контрактів (Contracts Limitation)
Будь-який учасник мережі може верифікувати транзакції. Щоб це було можливим, виконання контрактів мусить бути абсолютно **детермінованим** (результат має залежати *тільки* від вхідних даних та внутрішнього стану мережі).
Через це смарт-контракти **не можуть**:
- Мати прямий доступ до зовнішніх ресурсів інтернету (наприклад, викликати HTTP API погоди чи біржі).
- Генерувати справжні випадкові числа (truly random numbers), оскільки кожна нода має отримати ідентичний результат при повторному обчисленні.

**Рішення для зв'язку із зовнішнім світом:**
Для отримання даних (наприклад, курсу USD/ETH або надійного Random Number) використовують **Блокчейн Оракули (Blockchain Oracles)**. Найпросунутішим рішенням на сьогодні є мережа **Chainlink**.

## 5.4. Газ та Оплата обчислень (Paying for Computation)

В Ethereum ви платите за кожну виконану машиною інструкцію. Чим складніші обчислення — тим більший розмір оплати. Оплату збирають вузли-валідатори (завдяки поточному консенсусу Proof-of-Stake), які і надають свою обчислювальну інфраструктуру. Окрім винагороди, це також фундаментальний захист мережі від хакерів (неможливо запустити нескінченний цикл безкоштовно).

### Газ (Gas)
Комісія розраховується не безпосередньо в ефірі (Ether), а у спеціальній віртуальній одиниці **"газ" (Gas)**.
- Газ не можна купити, продати або зберігати (це лише одиниця вимірювання роботи).
- Кожна команда (opcode) має фіксовану вартість у газі.
- Відправник кожної транзакції вказує обмінний курс (Gas Price — скільки ETH він готовий платити за 1 одиницю газу).

*Вартість деяких команд (наприклад):*
- `ADD` / `LT` / `EQ` – 3 газу (базові операції).
- `MUL` – 5 газу.
- `SHA3` – від 30 газу (хешування дороге).
- `CREATE` – 32,000 газу (розгортання контракту вкрай дороге).
- `SSTORE` – до 20,000 газу (запис даних у блокчейн є **найдорожчою** операцією EVM).

### Відшкодування або втрата газу (Gas and Refund)
Коли ви відправляєте транзакцію, ви задаєте ліміт `Max gas to spend` (Gas Limit).
- **Успіх (Enough gas / Success):** Функція виконала всі інструкції. Зміни стану успішно збережені. Весь *невикористаний* газ конвертується назад в ефір і **повертається** на ваш гаманець (Remaining Ether refunded).
- **Невдача (Not enough gas / Failure):** Функція перервалася посередині виконання через брак ліміту газу. Стан контракту відкочується назад (ніби виклику і не було). Однак **увесь витрачений до моменту зупинки газ згорає** (не відшкодовується), оскільки мережа виконала цю роботу.

## 5.5. Транзакції проти Викликів (Transactions and Calls)

Існує критична алгоритмічна різниця між записом та читанням.
1. **Транзакція (Transaction):** Змінює стан (State) Ethereum мережі. Коштує реального ефіру (Ether). Потребує часу (кілька секунд) для перевірки та запису у блок.
2. **Виклик (Call):** **Не змінює станів**. Будучи лише "чистим читанням", вони **безкоштовні** (Free) та набагато швидші. Зазвичай обробляються безпосередньо вузлом (RPC-провайдером), до якого ви звернулися.

### Модифікатори функцій (Function Modifiers)
У Solidity є спеціальні ключові слова, які позначають такі безкоштовні функції `view` та `pure`:
```solidity
contract Foo {
    uint public val;
    
    // Транзакція (змінює стан, потребує газу та часу на блок)
    function setVal(uint newVal) public { 
        val = newVal; 
    }
    
    // Виклик (Reads from state. Безкоштовний для зовнішнього світу)
    function getVal() public view returns(uint) {
        return val;
    }
    
    // Виклик (Does not read state. Безкоштовний, суто обчислення аргументів)
    function sum(uint a, uint b) public pure returns(uint) {
        return a + b;
    }
}
```

## 6. Покрокова інструкція: Розгортання контракту в Remix (Deploying Contract with Remix)

Щоб написати та протестувати смарт-контракт, вам не обов'язково розгортати локальний сервер. Ми використовуватимемо [Remix IDE](https://remix.ethereum.org/) — потужне онлайн-середовище.

**Крок 1: Написання коду**
1. Відкрийте Remix і створіть файл у папці `contracts` (наприклад, `Greeter.sol`).
2. Вставте один із базових кодів із наших прикладів (наприклад, `HelloWorld_v1.sol`).
3. Використовуючи вкладку **Solidity Compiler** (іконка з буквою S), скомпілюйте код (кнопка *Compile Greeter.sol*).

**Крок 2: Розгортання (Deploy) на тестнет**
1. Перейдіть до вкладки **Deploy & Run Transactions**.
2. У полі **Environment** виберіть `Injected Provider - MetaMask`.
   *Архітектура підключення:*
   - Ваш **Браузер** з плагіном **MetaMask** не є повноцінною блокчейн-нодою.
   - Замість цього MetaMask інкапсулює ваш приватний ключ і формує транзакцію, але відправляє її через Ethereum API до інфраструктурного провайдера (зазвичай це **INFURA** або Alchemy).
   - **INFURA** вже безпосередньо маршрутизує та передає цей запит до справжніх вузлів мережі (**Ethereum nodes**), які виконують її та закріплюють у блокчейні.
3. Після вибору MetaMask запропонує підключитися. Переконайтеся, що ви вибрали **тестову мережу** (Sepolia), де ефір безкоштовний.
4. Натисніть кнопку **Deploy**. Підтвердіть транзакцію в MetaMask. Це платна дія мережі (Transaction), тому спишеться трохи тестового газу.

**Крок 3: Взаємодія з контрактом (Interact)**
1. Після успішного деплою внизу з'явиться розгорнутий список **Deployed Contracts**.
2. Жовті або червоні кнопки — це **Транзакції** (змінюють стейт, наприклад `setVal`). Вони попросять підтвердження в MetaMask і зпишуть газ.
3. Сині кнопки — це **Виклики (Calls)** (наприклад, `getVal` або `sum`). Вони виділені синім кольором тому, що вони є `view`/`pure` і є повністю безкоштовними — результат з'явиться миттєво.

**Крок 4: Знищення контракту (Remove)**
Якщо у контракті реалізована функція `selfdestruct(owner)`:
1. Виклик цієї функції (через жовту/червону кнопку) знищить контракт у блокчейні назавжди і поверне залишки балансу вказаній адресі (owner).
2. Якщо такої функції немає (що є сучасним стандартом безпеки для багатьох архітектур), контракт назавжди залишається на адресі.
3. Щоб просто прибрати блок контракту з візуального інтерфейсу Remix, натисніть "хрестик" або кнопку кошика поруч із адресою контракту.

## 7. HelloWorld: стан і виклики

Приклад: `HelloWorld_v1.sol`.

```solidity
contract HelloWorld {
    uint private value;

    function setValue(uint newValue) public {
        value = newValue;
    }

    function getValue() public view returns (uint) {
        return value;
    }
}
```

Що тут важливо: контракт має **стан** — змінну `value`, що зберігається в storage. `setValue` змінює стан (коштує газ); `getValue` лише читає (view, не змінює стан, виклик без плати ззовні). Кожен виклик функції — це транзакція (або внутрішній виклик з іншої транзакції). Контракт не "крутиться" сам по собі, він реагує лише на виклики.

## 8. Selfdestruct і незворотність

Приклад: `HelloWorld_v2_SelfDestruct.sol` — додано функцію `remove()`, яка викликає `selfdestruct(payable(address(0)))`. Після виконання контракт перестає існувати, а залишок ETH (якщо був) відправляється на вказану адресу. Це незворотна операція: на ланцюгу її не відкотити.

## 9. Voter: конструктор, масиви, перевірки

Приклад: `Voter_v1.sol`.

- **constructor(string[] memory _options):** викликається один раз при деплої. Ініціалізує список варіантів і масив голосів.
- **vote(uint option):** збільшує лічильник для обраного варіанту. `require(0 <= option && option < options.length, "Invalid option")` — перевірка введення; при невиконанні транзакція відкочується (revert).
- **getOptions / getVotes:** читання даних (view).

Концепти: контракт як "об’єкт" зі станом; перевірки через `require`; газ за запис (votes[option] += 1) платить той, хто викликає `vote`.

---

## 10. Один голос на адресу: mapping

Приклад: `Voter_v2_OneVoteOnly.sol`.

- `mapping(address => bool) hasVoted` — для кожної адреси зберігаємо, чи вже голосувала.
- У `vote`: `require(!hasVoted[msg.sender], "Already voted")`; після перевірки викликається `recordVote(option)`, де встановлюється `hasVoted[msg.sender] = true` і оновлюється масив голосів.

Концепти: `msg.sender` — адреса, що викликала функцію; mapping для "один голос на адресу"; порядок: спочатку перевірки, потім зміна стану (підхід Checks-Effects-Interactions).

---

## 11. Голосування за рядком і вартість газу

У `Voter_v3_StringOptions.sol` є перевантаження `vote(string memory option)`: порівняння рядків через `keccak256(bytes(a)) == keccak256(bytes(b))`, цикл по всіх options. Це дорожче за газ, ніж голосування за індексом (uint). У `Voter_v4_VoteByString.sol` використовується `mapping(string => OptionPos) posOfOption`, заповнений у конструкторі: пошук за рядком стає O(1) замість циклу — приклад оптимізації газу через структуру даних.

## 12. Безпека: Reentrancy і Checks-Effects-Interactions

Небезпечно спочатку відправляти кошти зовні, а потім оновлювати внутрішній стан (атака Reentrancy). Щоб цьому запобігти, застосовують патерн Checks-Effects-Interactions (спочатку перевірки, потім зміна стану, і лише в кінці — зовнішні виклики). Детальний розбір цього патерну на реальному коді дивіться у практикумі [p01: Безпека та CEI](workshops/p01_security_checks_effects_interactions.md) та у [Розборі Кейсів (The DAO Hack)](case_studies.md).

## 14. Сучасні стандарти розробки (Modern Development Standards)

З розвитком екосистеми Ethereum підходи до розробки значно змінилися. В сучасному production-середовищі варто враховувати такі аспекти:

### 1. Layer 2 (L2) рішення та масштабованість
Оскільки комісії (газ) у головній мережі Ethereum (L1) стали занадто дорогими для звичайних користувачів, більшість сучасної розробки та деплою DApps відбувається на мережах другого рівня (таких як **Arbitrum**, **Optimism**, **zkSync**, **Base**). Важливо розуміти:
- Код на Solidity компілюється та працює на L2 абсолютно так само, як і на L1 (завдяки EVM-сумісності).
- Комісії за транзакції там коштують долі центів, тоді як на L1 можуть сягати десятків доларів.

### 2. Професійні фреймворки (Foundry та Hardhat)
**Remix IDE**, який ми використовували вище, є ідеальним інструментом для навчання або швидкого прототипування. Однак у реальних проектах архітектура розгортається локально за допомогою спеціалізованих фреймворків:
- **Foundry** — сучасний галузевий стандарт (написаний на Rust). Дозволяє писати складні тести безпосередньо на Solidity, має неймовірну швидкість компіляції та виконання тестів.
- **Hardhat** — популярний JavaScript/TypeScript фреймворк, який дозволяє розгортати локальні форки (копії) реальної мережі Ethereum для тестування контрактів зі збереженням стейту.

### 3. Оновлювані смарт-контракти (Proxy Patterns)
За замовчуванням деплой контракту незмінний — після відправлення у блокчейн його логіку не можна переписати. Щоб обійти це обмеження і дозволити виправлення багів (оновлення коду), сучасний стандарт вимагає використання **Proxy-патернів** (наприклад, Transparent Proxy або UUPS). 
- Створюються два контракти: `Proxy` (який зберігає стан/баланси та адресу користувача) та `Implementation` (який містить логіку).
- Коли потрібне оновлення, власник просто направляє вказівник `Proxy` на нову адресу `Implementation`.

### 4. Стандарти токенів
Сучасний курс по Web3 неможливий без базового розуміння стандартів токенів (протоколів ERC), що лежать в основі DeFi та NFT:
- **ERC-20** — стандарт взаємозамінних (fungible) токенів (наприклад, USDT, стейблкойни). Всі токени однакові.
- **ERC-721** — стандарт невзаємозамінних токенів (NFT). Кожен токен є унікальним (картини, цифрова нерухомість).
- **ERC-1155** — мультитокенний стандарт (часто використовується в іграх для одночасного зберігання як унікальних, так і звичайних предметів у межах одного смарт-контракту).

## Питання та відповіді (екзаменаційний блок)

**П1.** Хто платить за виконання функції смарт-контракту і як це пов’язано з газом?

**В1.** За виконання платить той, хто ініціював транзакцію (викликав функцію): він задає gas limit і gas price. Витрачений газ списується з його рахунку. Якщо газу не вистачило, виконання відкочується (revert), а вже витрачений газ не повертається.

---

**П2.** Навіщо в Solidity використовують `require`? Що відбувається, якщо умова не виконується?

**В2.** `require(умова, "повідомлення")` перевіряє умову перед подальшим виконанням. Якщо умова false, поточна транзакція негайно переривається (revert): зміни стану скасовуються, повідомлення може бути повернене викликачу. Це захист від невалідних вводів і порушення інваріантів контракту.

---

**П3.** Що таке `msg.sender` і як його використовують для "один голос на адресу"?

**В3.** `msg.sender` — адреса (акаунт або контракт), яка безпосередньо викликала поточну функцію. Для обмеження "один голос на адресу" зберігають mapping (наприклад, `mapping(address => bool) hasVoted`) і перед зарахуванням голосу перевіряють `require(!hasVoted[msg.sender])`, після чого встановлюють `hasVoted[msg.sender] = true`.

---

**П4.** У чому різниця між функціями view/pure і функціями, що змінюють стан?

**В4.** `view`: функція лише читає стан, не змінює його; виклик ззовні не потребує транзакції (не сплачується газ). `pure`: не читає і не змінює стан, лише обчислення. Функції без view/pure можуть змінювати storage; їх виклик — це транзакція, що сплачується газом.

---

**П5.** Що таке reentrancy і який патерн захищає від нього?

**В5.** Reentrancy — випадок, коли під час виконання вашого контракту ви робите зовнішній виклик (наприклад, переказ ETH), а контракт-одержувач у callback знову викликає вашу функцію. Якщо стан оновлено після переказу, він може ще мати "старі" значення, і атакуючий може повторно отримати кошти. Захист: патерн Checks-Effects-Interactions — спочатку оновити внутрішній стан, потім робити зовнішні виклики; можна використовувати модифікатори типу nonReentrant.

## 🎓 Інженерний міні-кейс (на 1 бал)

Ви проводите аудит смарт-контракту біржі. Ви бачите такий код функції виведення коштів:
```solidity
function withdraw(uint amount) public {
    require(balances[msg.sender] >= amount, "Not enough funds");
    msg.sender.call{value: amount}("");
    balances[msg.sender] -= amount;
}
```
**Питання:** Яке фундаментальне правило безпеки тут порушено і як називається атака, яка дозволить хакеру вивести всі гроші з контракту?

<details markdown="1">
<summary>Відповідь</summary>
Порушено патерн **Checks-Effects-Interactions**: стан (баланс користувача) змінюється *після* зовнішнього виклику (`msg.sender.call`). Атака називається **Reentrancy**. Хакер може створити контракт, який у функції `receive` знову викликає `withdraw`, поки баланс ще не зменшився, і так викачати весь ефір з біржі.
</details>

## 15. Побудова dApps: Клієнтська частина та інтеграція

Смарт-контракт — це лише бекенд. Для користувача "додатком" є веб-сайт або мобільна аплікація, яка взаємодіє з блокчейном. Ця взаємодія відбувається через спеціальні бібліотеки та гаманці.

### Архітектурний патерн (Provider / Signer / Contract)
Більшість сучасних dApps використовують три основні абстракції (на прикладі бібліотек `ethers.js` або `viem`):

1.  **Provider (Провайдер):** Це "вікно" у блокчейн. Він дозволяє тільки читати дані (наприклад, баланс, стан контракту, номер блоку). Провайдер підключається до ноди (через Infura, Alchemy або сам гаманець).
2.  **Signer (Підписант):** Це абстракція вашого гаманця (Account). Він має доступ до приватних ключів і може підписувати транзакції. Будь-яка дія, що змінює стан блокчейну (запис), потребує Signer.
3.  **Contract (Об'єкт контракту):** Програмний об'єкт, що поєднує адресу контракту, його ABI (інтерфейс) та Провайдера/Підписанта. Дозволяє викликати функції контракту як звичайні JavaScript-методи.

### Інтеграція з гаманцем (EIP-1193)
Браузерні гаманці (MetaMask, Rabby) ін'єктують об'єкт `window.ethereum` у веб-сторінку.
- **Підключення:** `window.ethereum.request({ method: 'eth_requestAccounts' })`.
- **Зміна мережі:** Гаманець повідомляє додаток про зміну мережі або акаунту через події (events). Розробник повинен обробляти це, щоб уникнути помилок (наприклад, відправка токенів не в ту мережу).

### ABI (Application Binary Interface)
ABI — це JSON-файл, який генерується компілятором Solidity. Він описує всі функції контракту, їхні аргументи та типи повернення. Без ABI фронтенд не знатиме, як правильно закодувати виклик функції для EVM.
